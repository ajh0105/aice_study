🧠 이 AI는 CNN 방식인가요?

정확히는 CNN + RNN 혼합 방식이에요. 한 줄로 요약하면:

"눈(CNN)으로 사진을 보고, 뇌(RNN)로 글자 순서를 읽는 AI"

  • CNN 역할 — 번호판 이미지를 스캔해서 선·모양·글자 패턴 같은 시각적 특징을 추출해요
  • RNN(BiLSTM) 역할 — CNN이 뽑은 특징을 앞뒤 양방향으로 읽으면서 글자 순서를 이해해요
  • CTC 역할 — 최종적으로 "12가3456" 같은 텍스트로 변환해서 출력해요

순수 CNN만으로는 글자 순서를 이해하기 어려워요. 번호판처럼 여러 글자가 순서대로 나열된 경우엔 앞뒤 관계를 기억하는 RNN이 반드시 필요해서, 이 모델은 CNN(특징 추출) + RNN(순서 이해) + CTC(텍스트 출력)을 조합한 구조예요.

🚗 이 AI는 무엇을 하나요?

도로에서 자동차 번호판 사진을 찍으면, AI가 그 사진을 보고 번호판에 쓰인 글자를 그대로 텍스트로 읽어내는 기술이에요. 예를 들어 사진에서 12가3456 이라고 읽어주는 거죠. 이 기술을 OCR(광학 문자 인식)이라고 해요.

한국 번호판에는 두 가지 형식이 있어요:
 • 7자리: 숫자 2개 + 한글 1개 + 숫자 4개 (예: 12가3456)
 • 8자리: 숫자 3개 + 한글 1개 + 숫자 4개 (예: 123가4567)

🔤 기초 AI 용어

AI (인공지능)
사람처럼 생각하고 판단할 수 있도록 만든 컴퓨터 프로그램이에요. 사람이 직접 규칙을 짜지 않아도, 수많은 예시를 보면서 스스로 규칙을 찾아내요.
딥러닝 (Deep Learning)
AI 학습 방법 중 하나예요. 사람의 뇌 신경망을 흉내 낸 "층(layer)"을 여러 겹 쌓아서, 데이터에서 복잡한 특징을 스스로 찾아내요. "깊은(deep) 학습"이라는 뜻이에요.
OCR
Optical Character Recognition의 약자예요. "광학 문자 인식"이라고도 해요. 사진이나 이미지 속 글자를 읽어서 텍스트로 바꿔주는 기술이에요.
모델 (Model)
학습을 마친 AI를 파일로 저장한 것이에요. 마치 시험을 다 치른 학생이 "졸업장"을 받는 것처럼, 학습된 AI의 지식을 파일(best.pt)에 저장해요.
학습 (Training)
AI에게 수많은 예시(번호판 사진 + 정답 텍스트)를 보여주면서 올바르게 읽는 방법을 가르치는 과정이에요. 학교에서 공부하는 것과 같아요!
에폭 (Epoch)
전체 학습 데이터를 한 번 다 보는 것을 "1 에폭"이라고 해요. 예를 들어 50 에폭 학습이면, 모든 사진을 50번씩 반복해서 공부하는 거예요.
정확도 (Accuracy)
AI가 100개의 번호판 중 몇 개를 맞혔는지 나타내요. 99.80%면 100개 중 99.8개를 정확히 읽는다는 뜻이에요. 숫자가 높을수록 좋아요!
손실 (Loss)
AI의 "틀린 정도"를 숫자로 나타낸 거예요. 정답에서 멀수록 손실이 크고, 정답에 가까울수록 손실이 작아져요. 학습 목표는 손실을 최대한 줄이는 것!
파라미터 (Parameter)
AI 내부의 수많은 숫자들이에요. 학습하면서 이 숫자들이 조금씩 바뀌면서 AI가 점점 똑똑해져요. 이 모델은 약 900만 개의 파라미터를 가지고 있어요.
학습률 (Learning Rate)
한 번 학습할 때 파라미터를 얼마나 크게 바꿀지 결정해요. 너무 크면 학습이 불안정하고, 너무 작으면 학습이 너무 느려요. 적절한 균형이 중요해요!
배치 (Batch)
한 번에 AI에게 보여주는 사진 묶음이에요. 배치 크기=32면 32장씩 묶어서 학습해요. 너무 크면 메모리가 부족하고, 너무 작으면 학습이 불안정해요.
GPU
그래픽 처리 장치예요. 게임 그래픽에 쓰이지만, AI 학습에도 아주 빠른 계산이 필요하기 때문에 GPU를 사용해요. 이 프로젝트에서는 RTX 4060 Laptop을 사용했어요.

🧠 모델 구조 용어

CNN (합성곱 신경망)
이미지에서 특징을 찾아내는 AI 구조예요. 사진을 여러 필터로 스캔하면서 선, 모서리, 글자 모양 등의 패턴을 찾아내요. 눈의 역할을 한다고 볼 수 있어요.
RNN / LSTM / BiLSTM
글자의 순서를 기억하는 AI 구조예요. "1", "2", "가", "3", "4", "5", "6" 처럼 글자가 이어지는 순서를 이해해요. Bi는 앞뒤 방향 모두 읽는다는 뜻이에요.
STN (공간 변환 네트워크)
기울어지거나 찌그러진 번호판을 자동으로 똑바로 펴주는 AI 모듈이에요. 비뚤어진 사진을 먼저 교정해주는 "교정기"라고 생각하면 돼요.
VGG 백본
이미지에서 특징을 뽑아내는 CNN 구조의 한 종류예요. 4단계(Stage)로 이미지를 점점 작게 만들면서 중요한 특징만 추려내요.
CBAM (어텐션 모듈)
"어디에 집중할지" 결정하는 모듈이에요. 사람이 글을 읽을 때 글자 부분에 집중하듯, AI도 번호판의 글자 부분에 더 집중하도록 도와줘요.
CTC (연결주의 시간분류)
번호판 사진에서 글자가 정확히 어디 있는지 몰라도 학습할 수 있게 해주는 특별한 방법이에요. "blank" 토큰을 이용해 글자 경계를 자동으로 찾아줘요.
Focal CTC
어려운 문제는 더 열심히 공부하도록 만든 개선된 CTC예요. 쉬운 번호판보다 어려운 번호판에 더 많은 "페널티"를 주어서 AI가 집중해서 배우게 해요.
소프트맥스 (Softmax)
AI가 여러 글자 중 하나를 선택할 때 각 글자의 확률을 계산해요. 모든 확률을 더하면 100%가 돼요. 가장 높은 확률의 글자를 최종 답으로 선택해요.

⚙️ 학습 기법 용어

증강 (Augmentation)
학습 데이터를 인위적으로 다양하게 변형하는 기법이에요. 사진을 기울이거나, 흐리게 하거나, 어둡게 해서 AI가 다양한 상황에 대비하도록 훈련시켜요.
EMA (지수이동평균)
여러 에폭의 모델 파라미터를 평균 내어 더 안정적인 모델을 만드는 기법이에요. 하지만 너무 높은 decay값(0.9995)을 쓰면 오히려 개선이 안 보이는 문제가 생겨요.
파인튜닝 (Fine-tuning)
이미 잘 학습된 모델을 기반으로 더 작은 학습률로 추가 학습하는 방법이에요. 처음부터 다시 학습하는 것보다 빠르고, 기존에 잘 하던 것을 잃지 않는 장점이 있어요.
커리큘럼 학습
처음엔 쉬운 문제부터, 나중엔 어려운 문제를 주는 학습 방법이에요. 마치 학교에서 1+1부터 시작해서 방정식을 배우는 것처럼, AI도 단계적으로 어려워지는 예제를 학습해요.
웜업 (Warmup)
학습 초반에 학습률을 서서히 높이는 기법이에요. 너무 빠르게 시작하면 AI가 흔들리기 때문에, 처음엔 천천히 적응할 시간을 주는 거예요.
Early Stopping (조기 종료)
성능이 더 이상 개선되지 않으면 학습을 미리 멈추는 기법이에요. 과학습(overfitting)을 방지하고, 불필요한 학습 시간을 줄일 수 있어요.
과학습 (Overfitting)
AI가 학습 데이터만 너무 외워서, 새로운 데이터에는 틀리는 현상이에요. 시험 문제를 달달 외웠지만 조금 다른 응용 문제는 못 푸는 것과 비슷해요.
WeightedRandomSampler
드문 데이터를 더 자주 뽑아서 학습시키는 방법이에요. '희귀 글자(바, 배, 사 등)'가 적게 있으면 AI가 이를 못 배울 수 있어서, 이런 데이터를 4배로 더 많이 보여줘요.
ECE (보정 오차)
AI가 "나는 90% 확신해"라고 할 때 실제로 90% 확률로 맞는지 측정해요. ECE가 낮을수록 AI의 자신감이 실제 정확도와 일치하는, "솔직한 AI"예요.
FP32 / FP16 (부동소수점 정밀도)
AI 내부 숫자를 저장하는 방식이에요. FP32(32비트, 4바이트)는 정밀하지만 무겁고, FP16(16비트, 2바이트)은 절반 크기로 GPU에서 2배 빠르게 계산해요. 학습은 FP32로, 배포(추론)는 FP16으로 변환하면 모델이 가벼워지고 속도가 빨라져요.
Tensor Core
NVIDIA GPU 안에 있는 특별한 연산 유닛이에요. FP16 행렬 계산을 FP32보다 최대 8배 빠르게 처리해요. RTX 4060에도 탑재되어 있어서 FP16 변환만 해도 추론 속도가 크게 올라가요.
AMP (자동 혼합 정밀도)
PyTorch가 자동으로 연산에 따라 FP32와 FP16을 골라서 쓰는 기능이에요. 무거운 계산(행렬곱)은 FP16으로 빠르게, 정밀도가 필요한 계산(소프트맥스)은 FP32로 안전하게 처리해요.
ONNX
Open Neural Network Exchange의 약자예요. 다양한 AI 프레임워크(PyTorch, TensorFlow 등) 간에 모델을 공유할 수 있는 표준 파일 형식이에요. best.pt → best.onnx로 변환하면 TensorRT 등 다양한 배포 도구를 쓸 수 있어요.
TensorRT
NVIDIA가 만든 AI 추론 최적화 엔진이에요. FP16 변환 + 연산 최적화 + 메모리 최적화를 모두 자동으로 해줘요. ONNX 파일을 TensorRT 엔진으로 변환하면 같은 GPU에서 최대 4~8배까지 빠른 추론이 가능해요.
그래디언트 클리핑
학습 중 파라미터가 너무 크게 변하지 않도록 제한하는 안전장치예요. 학습이 폭발적으로 불안정해지는 것을 막아줘요. GRAD_CLIP=2.0으로 설정했어요.

📊 평가 지표 용어

Val Accuracy (검증 정확도)
학습에 사용하지 않은 검증용 사진으로 측정한 정확도예요. AI가 새로운 사진도 잘 읽는지 확인하는 지표예요.
Hard Accuracy (어려운 테스트 정확도)
6가지 어려운 상황(작은 번호판, 흔들림, 야간, 잘린 번호판, 기울어짐, 희귀 글자)에서 측정한 정확도예요. 이 점수가 높아야 실제 도로에서도 잘 동작해요.
Rare Accuracy (희귀 글자 정확도)
데이터가 적은 희귀 한글(바, 배, 사, 아, 자, 허)이 들어간 번호판을 얼마나 잘 읽는지 측정해요.
Comp Score (종합 점수)
여러 지표를 가중 평균한 최종 점수예요. 공식: 0.5×어려운정확도 + 0.3×희귀정확도 + 0.2×검증정확도 − ECE패널티. 이 점수가 가장 높을 때 best.pt를 저장해요.
Conf Accuracy (신뢰도 정확도)
AI의 예측 신뢰도가 높은 경우(예: 확신도 80% 이상)에만 평가한 정확도예요.
Hard 6개 카테고리
S=작은번호판(Small), B=모션블러(Blur), N=야간(Night), C=잘린이미지(Crop), K=기울어짐(sKew), R=희귀글자(Rare). 각 카테고리 약 250장씩 테스트해요.

v1부터 ocrv6.5까지, 이 AI가 어떻게 탄생하고 성장했는지 알아봐요!

v1 · 시작
처음 만든 OCR — "일단 해보자!"

가장 단순한 규칙 기반 방식으로 시작했어요. AI가 스스로 배우는 게 아니라, 사람이 직접 "글자가 이런 모양이면 이거다"라고 규칙을 만들어준 거예요.

  • 방식: 규칙 기반(Template Matching) + 기본 이미지 처리
  • 결과: 깨끗한 번호판은 어느 정도 읽지만, 조금만 어두워지거나 기울어지면 틀림
  • 한계: 새로운 상황에 유연하게 대처하지 못함
정확도: ~70% 이하 (상황에 따라 크게 다름)
v2 · v3 · 개선 시도
딥러닝 도입 — "AI에게 직접 배우게 하자!"

규칙 기반의 한계를 느끼고 딥러닝을 도입했어요. 수천 장의 번호판 사진을 AI에게 보여주며 스스로 특징을 찾게 했어요.

  • v2: 기본 CNN으로 글자 분류 시작. 번호판을 7~8개 칸으로 나눠서 각각 인식
  • v3: 데이터 양을 늘리고 증강 기법 추가. 어두운 사진, 기울어진 사진도 학습
  • 여전히 한계: 글자가 붙어있거나 잘린 경우 오인식
정확도: ~85-90% 수준
v4 · 구조 혁신
CTC 도입 — "글자 경계를 AI가 직접 찾게 하자!"

v4에서 CTC(Connectionist Temporal Classification)를 도입했어요. 이전까지는 글자 하나하나의 위치를 직접 표시해줘야 했는데, CTC를 쓰면 전체 번호판 텍스트만 알려줘도 AI가 알아서 배워요.

  • CNN으로 특징 추출 → RNN으로 순서 이해 → CTC로 최종 텍스트 출력
  • 글자 경계 표시 작업(어노테이션) 부담 대폭 감소
  • 이어진 글자도 올바르게 분리해서 읽기 시작
정확도: ~93-95% 수준
v5 · 현재 구조 완성
V5OCR — 지금의 AI 구조가 완성되다!

v5에서 현재까지 사용하는 AI 구조(아키텍처)가 완성됐어요. STN → VGG 4단계 → CBAM → BiLSTM → CTC 파이프라인이 확정되었어요.

  • STN: 기울어진 번호판 자동 교정
  • VGG 4단계 백본: 이미지 특징 추출
  • CBAM 어텐션: 중요한 부분 집중
  • 2층 BiLSTM(256): 앞뒤 방향으로 글자 순서 이해
  • Linear(51): 51개 클래스(blank + 숫자 10개 + 한글 40개) 분류
  • Focal CTC 도입으로 어려운 글자 학습 강화
Val: 98.70% / Hard: 98.00% / Rare: 100.00% / Conf: 99.24%
ocrv6 · 기반 확립
ocrv6 — 한글 40자 완전 지원!

v5를 기반으로 더 많은 한글 글자를 지원하고, 학습 데이터를 대폭 확충했어요. 희귀 글자(바·배·사·아·자·허)에 대한 별도 처리도 시작됐어요.

  • 한글 40자 전체 지원 (가~히 중 번호판에 쓰이는 글자 모두)
  • 그룹 인식 기반 train/val 분리 (같은 번호판이 학습/검증에 둘 다 들어가는 오류 수정)
  • WeightedRandomSampler로 희귀 글자 3배 오버샘플링

ocrv6.3 · 균형 달성
ocrv6.3 — 검증&어려운 지표 동시에 높이다!

각 단계 자세히 보기

📐

1단계: STN (공간 변환)

사진이 기울어져 있거나 원근감이 있어도 자동으로 정면에서 찍은 것처럼 교정해줘요. 6개의 숫자(변환 행렬)로 이미지를 변형해요. 나머지 단계가 훨씬 쉽게 글자를 읽을 수 있게 돕는 "전처리 담당"이에요.

🔍

2단계: VGG 백본 (특징 추출)

4개의 단계(Stage)로 이미지를 점점 작게 만들면서 중요한 특징만 추려내요. Stage 1: 64채널 → Stage 2: 128채널 → Stage 3: 256채널 → Stage 4: 512채널. 각 단계마다 더 복잡한 패턴을 인식해요.

👁️

3단계: CBAM (어텐션)

채널 어텐션(어떤 특징이 중요한지)과 공간 어텐션(이미지의 어느 위치가 중요한지)을 계산해요. "지금은 이 부분의 이 특징에 집중해!" 라고 알려주는 안내자 역할이에요.

🧠

4단계: BiLSTM (순서 이해)

256 유닛짜리 양방향 LSTM 2층이에요. 왼쪽→오른쪽, 오른쪽→왼쪽 두 방향으로 모두 읽어서 앞뒤 문맥을 이해해요. "3 다음에 뭐가 오지?" 같은 순서 관계를 학습해요.

📊

5단계: Linear 분류기

Linear(512→51)로 51개 클래스 중 하나를 선택해요. 51개 = blank(1) + 숫자(10: 0-9) + 한글(40: 가~히 중 번호판 글자). 소프트맥스로 확률을 계산하고 가장 높은 확률의 글자를 선택해요.

📝

6단계: CTC 디코딩

blank 토큰을 제거하고 연속 중복을 합쳐서 최종 텍스트를 출력해요. 예: [blank, 1, 1, blank, 2, blank, 가, blank, 3, 4, 5, 6] → "12가3456". 글자 경계를 자동으로 찾아내는 마법 같은 기술!

증상: ocrv6.4 2차 학습에서 ep10부터 EMA 시작 후 rare가 0.9904 → 0.9809로 하락.

원인: EMA_DECAY=0.999로 설정했는데, ep10에서 EMA가 시작되면 처음 몇 에폭 동안 "무거운 초기 가중치"가 많이 반영되어 성능이 일시 저하됩니다. 또한 이전 에폭(ep10 이전)의 약한 모델 상태가 평균에 포함돼요.

해결: ocrv6.5에서 EMA 완전 제거. eval_model = model로 그대로 평가. 파인튜닝에서는 LR=5e-5로 이미 안정적이므로 EMA 불필요.

TS-3 6 에폭 동안 지표가 전혀 변하지 않아요! 지표 동결 ▼

증상: ocrv6.5 초기 버전(EMA 포함)에서 ep1~ep6 모두 val=0.9892, rare=0.9904, comp=0.9890으로 완전히 동일.

원인 — EMA decay 수학:

  • EMA_DECAY = 0.9995 (ocrv6.5 초기 설정)
  • 6 에폭 후 새 가중치 반영 비율 = 1 - 0.9995^6 ≈ 0.3%
  • 즉, 새로 학습한 내용이 0.3%밖에 반영이 안 되니 지표가 안 변하는 것!

해결: EMA 완전 제거. eval_model = model (raw model)로 직접 평가. 즉시 지표 개선 확인.

TS-4 "&는 예상되지 않았습니다" 오류가 나요! 환경 오류 ▼

증상: cmd.exe에서 & "C:\cuda\cuda\Scripts\python.exe" scripts/train.py 실행 시 "&는 예상되지 않았습니다" 오류.

⚡ FP16 변환 — 더 빠르게, 더 가볍게

🔢 FP32 vs FP16 — 왜 변환하나요?

학습은 FP32(32비트)로 해서 수치가 정확해요. 하지만 배포(추론)할 때는 FP16(16비트)으로 바꾸면:

  • 모델 크기: 31.8 MB → 약 16 MB (50% 절감)
  • 추론 속도: 최대 2배 향상 (GPU Tensor Core 활용)
  • 정확도 손실: 거의 없음 (<0.01%p) — 번호판 OCR은 51개 클래스 분류라 FP16 오차가 답을 바꾸지 않음

방법 A: model.half() — 가장 단순한 변환

# FP16으로 변환 후 추론
device = torch.device("cuda")
model = load_model("ocrv6.5/pt/best.pt", device)
model.eval()

# 모든 파라미터를 float16으로 변환
model_fp16 = model.half()

# 입력도 반드시 .half()로 맞춰야 함 (타입 불일치 시 오류)
x = preprocess(img_bgr).unsqueeze(0).to(device).half()
with torch.no_grad():
    lp = F.log_softmax(model_fp16(x).permute(1, 0, 2), dim=-1)
result = ctc_greedy(lp[:, 0, :])

# STN(grid_sample)이 불안정할 경우 STN만 FP32 유지
model.half()
model.stn.float()   # STN 서브모듈만 다시 FP32로

방법 B: AMP (자동 혼합 정밀도) — 실서비스 권장

# PyTorch AMP: 연산별로 FP16/FP32를 자동 선택
from torch.amp import autocast

model.eval()

def predict_amp(img_bgr):
    x = preprocess(img_bgr).unsqueeze(0).to(device)   # FP32 입력 그대로
    with torch.no_grad():
        with autocast(device_type="cuda", dtype=torch.float16):
            logits = model(x)                          # conv/matmul은 FP16
        # log_softmax는 FP32로 (언더플로우 방지)
        lp = F.log_softmax(logits.float().permute(1, 0, 2), dim=-1)
    return ctc_greedy(lp[:, 0, :])

# AMP 장점: 입력 타입 걱정 없음, STN 자동 FP32 유지

❓ Q&A — 자주 묻는 질문

이 내용을 다 읽으면 어떤 질문에도 막힘없이 대답할 수 있어요!

🔤 기초 개념

Q. OCR이 뭔가요? 어떻게 번호판을 읽나요? ▼
OCR은 Optical Character Recognition(광학 문자 인식)의 약자예요. 사진 속 글자를 인식해서 텍스트로 바꿔주는 기술이에요. 번호판 OCR은 ① 사진을 입력받아 → ② AI 신경망이 각 위치에서 글자 확률을 계산하고 → ③ CTC 디코딩으로 최종 텍스트를 출력해요. "12가3456" 같은 결과가 나오죠!
Q. 딥러닝과 일반 프로그래밍의 차이가 뭔가요? ▼
일반 프로그래밍은 사람이 직접 "이런 상황이면 이렇게 해"라는 규칙을 코드로 작성해요. 딥러닝은 수천~수만 개의 예시(사진+정답)를 보여주면 AI가 스스로 규칙을 찾아내요. 번호판이 어두워도, 기울어져도 스스로 적응하는 능력이 있어요!
Q. CTC가 뭔가요? 왜 필요한가요? ▼
CTC(Connectionist Temporal Classification)는 글자 위치를 모르는 상태에서도 학습할 수 있게 해주는 기법이에요. 번호판에서 "12가3456"이 어디서부터 어디까지인지 일일이 표시하지 않아도 전체 텍스트만 알면 AI가 스스로 경계를 찾아요. blank 토큰을 이용해 [1, 1, blank, 2, blank, 가 ...] → "12가3456"으로 변환해요.
Q. 파라미터가 뭐고, 왜 많을수록 좋은 건가요? ▼
파라미터는 AI 내부의 숫자들이에요. 이 숫자들이 학습을 통해 조금씩 바뀌면서 AI가 똑똑해져요. 많을수록 더 복잡한 패턴을 학습할 수 있지만, 너무 많으면 학습 데이터만 외워서 새 데이터에는 못 쓰는 "과학습" 문제가 생겨요. 이 모델은 약 900만 개로 균형을 맞췄어요.
Q. 에폭(Epoch)이 많으면 무조건 좋은 건가요? ▼
꼭 그렇지 않아요! 에폭이 너무 많으면 AI가 학습 데이터를 "외워버려서" 새로운 사진에는 오히려 틀리는 과학습이 발생해요. 그래서 Early Stopping(조기 종료)을 사용해요. ocrv6.5에서는 ep9이 가장 좋았고, ep24에 PATIENCE=15에 걸려서 학습을 멈췄어요.

🏗️ 모델 구조

Q. STN이 왜 필요한가요? ▼
실제 도로에서 찍은 번호판은 완벽하게 정면에서 찍히지 않아요. 카메라 각도, 차량 속도, 도로 굴곡 때문에 기울어지거나 원근감이 생겨요. STN(Spatial Transformer Network)이 이런 변형을 자동으로 교정해줘서 뒤의 CNN이 훨씬 쉽게 글자를 인식할 수 있게 해줘요.
Q. BiLSTM이 단방향 LSTM보다 나은 이유는? ▼
단방향 LSTM은 왼쪽→오른쪽으로만 읽어요. 하지만 번호판에서 어떤 글자가 올지는 앞뒤 문맥을 모두 봐야 알 수 있어요. 예를 들어 "12?3456"에서 가운데 글자를 맞추려면 앞의 "12"와 뒤의 "3456" 모두 필요해요. BiLSTM은 양방향으로 읽어서 더 풍부한 문맥 정보를 활용해요.
Q. CBAM 어텐션은 왜 추가했나요? ▼
번호판 이미지에는 배경, 테두리, 볼트 구멍 등 글자와 관계없는 부분이 있어요. CBAM은 "어떤 위치(공간 어텐션)의 어떤 특징(채널 어텐션)이 중요한가"를 학습해서 AI가 실제 글자 부분에 집중하도록 도와줘요. 불필요한 정보에 낭비하는 계산을 줄일 수 있어요.
Q. 왜 입력 이미지를 흑백(그레이스케일)으로 변환하나요? ▼
번호판 인식에서 글자의 형태가 중요하지, 색상은 중요하지 않아요. 흑백으로 변환하면 데이터 크기가 1/3로 줄어서 학습이 빠르고 메모리도 절약돼요. 또한 야간, 주간, 흐린 날 등 조명 변화에 덜 민감해져서 더 강인한 모델이 돼요.
Q. 왜 51개 클래스인가요? ▼
blank(1개) + 숫자 0-9(10개) + 한글 40자 = 51개예요. 한글 40자는 한국 번호판에 실제로 사용되는 글자들이에요. blank는 CTC에서 "글자 없음" 또는 "글자 경계"를 나타내는 특수 기호예요. blank의 인덱스는 반드시 0이어야 해요(CTC 규칙).

📊 성능과 평가

Q. Comp Score(종합 점수)는 어떻게 계산하나요? ▼
Comp = 0.5 × Hard_acc + 0.3 × Rare_acc + 0.2 × Val_acc − ECE_penalty 로 계산해요. 가중치가 이렇게 설정된 이유: Hard(어려운 상황)를 가장 중요하게(50%), Rare(희귀 글자)도 중요하게(30%), 일반 정확도는 참고용(20%)으로 봐요. best.pt는 이 점수가 가장 높을 때 저장돼요.
Q. ECE(보정 오차)가 뭔가요? ▼
Expected Calibration Error예요. AI가 "90% 확신한다"고 할 때 실제로 90%의 확률로 맞는지를 측정해요. ECE=0.0061이면 AI의 자신감과 실제 정확도의 차이가 0.61%라는 뜻이에요. ECE가 낮을수록 신뢰할 수 있는 AI예요.
Q. ocrv6.3이 Rare는 97.48%인데 ocrv6.5가 99.52%인 이유가 뭔가요? ▼
ocrv6.5에서 희귀 글자 오버샘플링을 3배→4배로 늘리고, FocalCTC가 이 글자들에 더 집중해서 학습했어요. 또한 파인튜닝 기반(ocrv6.4 ep8 BEST)으로 시작해서 이미 좋은 기반 위에서 더 세밀하게 조정할 수 있었어요.
Q. Hard 세트의 6개 카테고리는 왜 선택했나요? ▼
실제 도로 환경에서 OCR이 실패하는 주요 원인들을 분석해서 선택했어요: S(작은 번호판 - 멀리서 찍힘), B(모션 블러 - 빠른 차), N(야간 - 어두운 환경), C(잘린 이미지 - 가장자리 차량), K(기울어짐 - 카메라 각도), R(희귀 글자 - 데이터 부족). 이 6가지에서 100%라면 실제 도로에서도 잘 동작한다는 신뢰성을 가져요.
Q. v5가 Rare 100%인데 왜 ocrv6.5보다 낮게 평가되나요? ▼
v5는 Hard 전체 98.00%로 특히 B(모션블러), N(야간) 등에서 약점이 있어요. ocrv6.5는 Hard 100.00%로 모든 어려운 상황을 커버해요. Rare 100%만 봐선 v5가 좋아보이지만, 다른 Hard 카테고리가 부족해서 종합 점수(Comp)는 ocrv6.5가 훨씬 높아요.

⚙️ 학습 과정

Q. 파인튜닝이 처음부터 학습하는 것보다 왜 좋은가요? ▼
① 빠름: 좋은 기반에서 시작하므로 더 빨리 좋은 결과가 나와요. ② 안전: 이미 잘 하던 것을 잃지 않아요 (catastrophic forgetting 방지). ③ 효율: 적은 데이터로도 효과적으로 개선 가능. ocrv6.5는 ocrv6.4 ep8(이미 rare=99.04%)에서 시작해서 ep9만에 99.52%를 달성했어요!
Q. 커리큘럼 학습이 왜 효과적인가요? ▼
사람도 쉬운 것부터 배우는 게 효과적이듯, AI도 마찬가지예요. 처음부터 너무 어려운 사진(극심한 블러, 야간, 기울어짐)을 보여주면 AI가 혼란스러워해요. p1(쉬움)으로 기초를 익히고, p2→p3로 점점 어려워지면 더 체계적으로 학습할 수 있어요.
Q. EMA가 좋은 기법인데 왜 ocrv6.5에서 제거했나요? ▼
EMA는 처음부터 학습할 때 유용해요. 학습 중간에 흔들리는 모델을 평균 내어 안정적으로 만들어줘요. 하지만 파인튜닝(LR=5e-5)처럼 변화가 작을 때 EMA_DECAY=0.9995를 쓰면 새 학습이 0.3%밖에 반영 안 돼요. 결과: 지표가 수 에폭 동안 전혀 변하지 않는 "동결" 현상. 이미 안정적인 파인튜닝에서는 EMA가 오히려 방해가 됐어요.
Q. WeightedRandomSampler가 왜 필요한가요? ▼
데이터가 불균형하면 AI는 많은 데이터의 글자만 잘 배우게 돼요. '가', '나', '다'가 각각 1000개인데 '바', '배'가 각각 50개라면, AI는 '가나다'만 잘 배우고 '바배'는 못 배워요. WeightedRandomSampler로 희귀 글자를 4배 더 자주 뽑아서 AI가 골고루 배우도록 해요.
Q. Focal CTC가 일반 CTC와 다른 점은? ▼
일반 CTC: 모든 샘플에 동일한 가중치로 학습. Focal CTC: 쉬운 샘플(loss 작음)은 가중치 낮게, 어려운 샘플(loss 큼)은 가중치 높게. 즉, "이미 잘 하는 건 조금만 공부하고, 못 하는 건 더 집중해서 공부하자!" 는 전략이에요. γ=1.2로 설정해서 어려운 번호판에 더 집중하게 해요.

🔧 실제 사용과 환경

Q. YOLO와 ocrv6.5 중 어느 것이 더 중요한가요? ▼
둘 다 중요해요! YOLO는 "번호판이 어디 있나"를 찾고, ocrv6.5는 "번호판에 뭐라고 쓰여 있나"를 읽어요. YOLO가 번호판 위치를 잘못 찾으면 OCR이 좋아도 소용없고, OCR이 약하면 YOLO가 잘 찾아도 글자를 틀리게 읽어요. 두 모델이 협력해야 완전한 시스템이 돼요.
Q. Windows에서 NUM_WORKERS를 0으로 해야 하는 이유는? ▼
Windows에서는 Python의 멀티프로세싱이 Linux/Mac과 다르게 동작해요. DataLoader의 NUM_WORKERS를 4 이상으로 설정하면 자식 프로세스 생성 방식 차이 때문에 오류나 데이터 로딩 문제가 생겨요. NUM_WORKERS=0이면 메인 프로세스에서 직접 데이터를 로드하므로 안전해요. 속도는 조금 느리지만 안정성이 훨씬 높아요.
Q. GPU가 없어도 사용할 수 있나요? ▼
네! GPU가 없어도 CPU로 추론(inference)은 가능해요. torch.load("best.pt", map_location="cpu")로 로드하면 돼요. 단, 학습(training)은 GPU 없으면 몇 시간이 아닌 며칠이 걸릴 수 있어요. 추론 속도도 GPU 대비 10-50배 느릴 수 있어요.
Q. best.pt 파일이 31.8 MB인 이유가 뭔가요? ▼
모델의 모든 파라미터(약 900만 개)를 FP32(float32, 4바이트/값)로 저장하기 때문이에요. 900만 × 4바이트 = 약 36 MB에서 PyTorch 최적화로 31.8 MB가 됐어요. FP16으로 변환하면 약 16 MB로 줄일 수 있고, 이 모델에서 정확도 손실은 <0.01%p로 거의 없어요.
Q. FP16 변환이 뭔가요? 왜 하나요? ▼
FP32(32비트)로 저장된 모델을 FP16(16비트)으로 바꾸는 것이에요. 학습은 수치 안정성을 위해 FP32로 하지만, 배포할 때는 FP16으로 바꾸면 4가지 이점이 있어요:

① 속도: GPU Tensor Core가 FP16 연산을 FP32보다 최대 2배 빠르게 처리
② 크기: 31.8 MB → 약 16 MB (50% 절감)
③ 배치 크기: 같은 VRAM으로 2배 더 많은 이미지 동시 처리 가능
④ 정확도: 번호판 OCR처럼 클래스 간 확률 차이가 큰 경우엔 FP16 오차가 정답을 바꾸지 않아요.
Q. FP16 변환 방법은 어떤 게 있나요? ▼
3가지 방법이 있어요:

① model.half(): 한 줄로 끝나는 가장 단순한 방법. 단, 입력도 반드시 .half()로 변환해야 해요.
② AMP (autocast): PyTorch가 자동으로 연산별로 FP16/FP32를 골라줘요. 입력을 FP32로 줘도 되고, 불안정한 연산(STN의 grid_sample 등)은 알아서 FP32로 유지해줘서 실서비스에 가장 안전해요.
③ ONNX FP16: PyTorch → ONNX 변환 후 FP16으로 바꾸는 방법. TensorRT와 연계해서 가장 빠른 배포가 가능해요. 사용법 페이지의 코드 예시를 참고하세요!

주의: FP16은 CUDA GPU 전용! CPU에서는 오히려 느려질 수 있어요.
Q. FP16 변환 시 주의할 점이 있나요? ▼
3가지 주의사항이 있어요:

① STN(grid_sample) 불안정: model.half()를 쓸 때 STN의 grid_sample 연산이 FP16에서 오차가 생길 수 있어요. 해결: model.stn.float()로 STN만 다시 FP32로 유지하거나, AMP 방식을 사용하세요.

② CTC log_softmax 언더플로우: log_softmax는 매우 작은 음수값을 다뤄서 FP16에서 -inf가 될 수 있어요. 해결: logits.float()로 FP32로 변환 후 log_softmax를 적용하세요.

③ CPU 환경 금지: CPU는 FP16 SIMD 지원이 제한적이라 오히려 FP32보다 느려요. device.type == "cuda"일 때만 FP16을 적용하세요.
Q. 모델을 더 가볍게 만들 수 있나요? ▼
FP16 외에도 여러 방법이 있어요: ① INT8 양자화(Quantization): float32→int8, 4배 가벼움. TensorRT에서 지원하지만 정확도 손실이 FP16보다 커요. ② 가지치기(Pruning): 덜 중요한 파라미터를 0으로 만들어 제거. ③ 지식 증류(Knowledge Distillation): 큰 모델이 작은 모델을 가르쳐서 작은 모델에 성능 이전. 모두 정확도와 속도 간의 트레이드오프가 있어요. 이 모델의 경우 FP16만으로도 충분한 개선이 가능해요.
Q. 어떤 상황에서 OCR이 실패할 수 있나요? ▼
Hard 6개 카테고리가 모두 100%이지만 극단적인 경우는 여전히 어려워요: ① 번호판이 진흙/눈으로 완전히 덮인 경우 ② 카메라 렌즈에 물방울이 가득 찬 경우 ③ 번호판이 심하게 파손되거나 가려진 경우 ④ 학습 데이터에 없는 외국 번호판 형식 ⑤ 해상도가 극히 낮아(8×32 이하) 글자 형태 자체가 손실된 경우.
Q. 앞으로 더 개선할 수 있는 방향은? ▼
ocrv6.5가 Hard 100%를 달성했지만 여전히 개선 방향이 있어요: ① Transformer 어텐션 도입(ViT 기반) ② TensorRT/ONNX로 추론 속도 최적화 ③ 더 많은 야간/악천후 데이터 수집 ④ 다양한 지역 번호판(특수 번호판 등) 지원 ⑤ 실시간 스트림 처리 최적화. 99.80%에서 99.90%+를 목표로 할 수 있어요.
