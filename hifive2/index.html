문서 한 줄 정의

핵심 관점

Jetson은 현장 통행 이벤트 생성, 중앙 서버는 저장·검증·조회, GPS/KPS는 위치 정합성 검증 데이터로 역할을 분리한다.

01

문서 목적

시연 순서가 아니라 산출물 기준으로 프로젝트 전체를 설명하기 위한 문서.

이 문서는 특정 시연 장면을 설명하기 위한 자료가 아니라, 프로젝트 결과물을 기준으로 시스템 목적, 구성, 파이프라인, 기술 요소, 데이터 구조, 기대효과를 통합 정리한 산출물이다.

이해 대상

프로젝트를 처음 보는 검토자, 현업자, 발표 평가자, 인수인계 담당자가 빠르게 구조를 파악할 수 있도록 한다.

문서 범위

Jetson Edge AI, 중앙 서버, 데이터 저장, GPS/KPS 위치 정합성, 운영 조회 구조를 포함한다.

제외 범위

Q&A성 방어 문장, 멘토 대화 내용, 해외 정책 상세, 단순 시연 순서 중심 설명은 제외한다.

02

30초 요약

처음 보는 사람이 가장 먼저 이해해야 하는 프로젝트 핵심.

HI-FIVE는 Jetson Edge AI가 차량과 번호판을 인식해 통행 이벤트를 생성하고, 중앙 서버가 GPS/KPS 위치 데이터와 정합성을 검증하는 스마트톨링 시스템이다. 핵심은 현장 추론, 이벤트 중심 데이터화, 중앙 검증, 위치 기반 확장성이다.

현장

Jetson에서 차량·번호판을 인식하고 OCR 결과를 누적한다.

이벤트

번호판, 시간, 차로, 신뢰도 기반 통행 이벤트를 생성한다.

검증

중앙 서버에서 GPS/KPS 위치 데이터와 정합성을 판단한다.

운영

Dashboard와 Chatbot으로 통행 이벤트와 상태를 조회한다.

03

프로젝트 개요

기존 하이패스 체계를 존중하면서 미래형 스마트톨링 확장 구조를 제안.

본 프로젝트는 기존 하이패스 체계를 부정하거나 대체하려는 목적이 아니다. 향후 정밀 위치 기반 스마트톨링 환경에서 필요해질 수 있는 차량 인식 자동화, 위치 정합성 검증, 이벤트 중심 데이터 관리, 장애 대응형 통신을 하나의 시스템 구조로 검토한 프로젝트이다.

핵심 방향: 중앙 서버가 AI 추론까지 모두 담당하는 구조는 차로·장비 수가 증가할수록 중앙 연산 자원과 응답성 부담이 커질 수 있다. 본 프로젝트는 현장 Edge에서 1차 통행 이벤트를 생성하고, 중앙 서버는 저장·검증·운영 판단에 집중하도록 역할을 분리한다.

기술 목표

  • Jetson 기반 차량·번호판 인식
  • OCR 후보 누적 및 Best-Fit 판정
  • 통행 이벤트 생성 및 중앙 서버 전송
  • GPS/KPS 위치 정합성 검증
  • 장애 대응형 통신 구조 구성

운영 목표

  • 통행 이벤트의 구조화된 저장
  • 위치 검증 결과의 상태 관리
  • 운영자 Dashboard 조회
  • Chatbot 기반 이벤트 검색
  • 향후 KPS 연계 확장성 확보
04

한눈에 보는 시스템 구조

각 구성요소가 담당하는 역할을 먼저 분리해서 이해한다.

Camera

차량 통행 영상 입력. 실시간 카메라 또는 사전 영상 입력 가능.

Jetson

YOLO/OCR/Best-Fit으로 통행 이벤트를 생성하는 현장 Edge AI 장비.

Ingress

Jetson 이벤트를 수신하고 스키마 검증 및 Backend 전달 수행.

Backend/Data

통행 이벤트 저장, GPS/KPS 위치 정합성 검증, 상태 관리.

Operation

Dashboard와 Chatbot으로 이벤트, 위치 검증, 장비 상태 조회.

역할 분리 원칙

FastAPI Ingress 이벤트 수신, 형식 검증, 전송 로그, Backend 전달 Transmission Log Edge와 업무 서버 사이의 수신 계층

중앙 서버 처리

Edge에서 생성된 이벤트를 운영 가능한 데이터로 검증·저장·조회하는 계층.

중앙 서버는 AI 추론을 대신하는 계층이 아니라, Edge에서 생성된 이벤트를 운영 가능한 데이터로 검증·저장·조회하는 계층이다.

01 Event Receive Jetson에서 생성된 통행 이벤트 수신
02 Validation 필수 필드, 장비 식별자, 이벤트 형식 검증
03 Save passage_event, OCR 후보, 전송 로그 저장
04 Location Match GPS/KPS 위치 데이터와 시간·거리 정합성 검증
05 Operation API Dashboard와 Chatbot 조회 API 제공

이벤트 기준

event_uuid, device_id, lane_id, plate_text, event_time, confidence

위치 기준

recorded_at, latitude, longitude, accuracy_m, toll_zone radius

상태 기준

match_result, event_status, needs_review, transmission_status

08

GPS/KPS 위치 정합성 검증

GPS/KPS는 결제수단이 아니라 통행 이벤트의 위치 정합성을 검증하기 위한 데이터.

역할 구분: Jetson은 영상 기반 통행 이벤트를 생성한다. GPS/KPS 위치 데이터는 중앙 Backend/Data 영역에서 통행 이벤트와의 위치 정합성 검증에 사용된다.

GPS

현재 구조에서 위치 데이터 또는 Replay 데이터로 활용한다. 통행 이벤트와 시간·거리 기준으로 매칭한다.

KPS

향후 국내 정밀 위치 인프라 도입 시 같은 구조에서 더 정밀한 위치 검증으로 확장할 수 있는 요소이다.

정합성 검증

위치 데이터와 영상 기반 통행 이벤트가 같은 시간·구역 기준에서 일치하는지 판단한다.

검증 기준

입력 데이터
Passage Event event_time plate_text lane_id + GPS/KPS Location recorded_at lat/lon accuracy_m
검증 로직
time_diff_ms distance_m toll_zone radius accuracy_m plate consistency
결과
ACCEPT REVIEW REJECT match_result 저장

정합성 상태

ACCEPT

영상 기반 통행 이벤트와 위치 데이터가 기준 시간·거리 범위 내에서 정상적으로 일치한 상태.

REVIEW

시간, 거리, 위치 정확도, OCR 신뢰도 등 일부 기준에서 추가 확인이 필요한 상태.

REJECT

위치 또는 시간 기준상 통행 이벤트와 위치 데이터의 정합성이 실패한 상태.

09

데이터 구조

통행 이벤트를 중심으로 OCR, 위치, 전송, 정산 후보 데이터가 연결된다.

테이블 역할 주요 데이터
passage_event 전체 시스템의 기준이 되는 통행 이벤트 event_uuid, device_id, lane_id, plate_text, event_time, event_status
plate_ocr_candidate 다중 프레임 OCR 후보 결과 저장 candidate_text, confidence, frame_index, track_id
event_transmission_log Edge → Ingress → Backend 전송 상태 기록 event_uuid, status, sent_at, received_at, retry_count
location_telemetry GPS/KPS 위치 데이터 또는 Replay 로그 저장 vehicle_id, recorded_at, latitude, longitude, accuracy_m
location_event_match 통행 이벤트와 위치 데이터의 정합성 검증 결과 event_uuid, time_diff_ms, distance_m, match_result
settlement_candidate 향후 정산 후보로 확장 가능한 이벤트 관리 event_uuid, candidate_status, amount_policy, review_required
데이터 모델 핵심: passage_event가 중심 테이블이며, OCR 후보, 위치 로그, 정합성 검증 결과, 전송 로그가 event_uuid를 기준으로 연결된다. 이를 통해 이벤트 추적, 검증, 운영 조회, 고도화된 정산 후보 관리가 가능하다.

Zero Copy 표현 기준: 본 프로젝트에서 Zero Copy는 전체 프로그램의 모든 구간에서 복사가 없다는 의미가 아니라, GStreamer/NVMM/CUDA Memory 기반으로 입력·전처리·추론 구간의 메모리 이동을 최소화하는 구조를 의미한다.

11.1 Jetson Orin Nano 8GB 기준 제원

GPU

Ampere Architecture 기반 GPU, 1024 CUDA Cores, 32 Tensor Cores 기준으로 Edge AI 추론을 수행한다.

CPU / Memory

6-core Arm Cortex-A78AE CPU와 8GB LPDDR5 통합 메모리를 활용한다. 발표자료의 “8GB GPU Memory”는 Jetson 통합 메모리 기준으로 정리한다.

Edge AI 기준

저전력 현장 AI 장비로, 차량·번호판 추론과 통행 이벤트 생성을 현장에서 처리하기 위한 PoC 기준 장비이다.

항목 Jetson Orin Nano 8GB 기준 본 프로젝트에서의 의미

CUDA Cores 1024 CUDA Cores 영상 전처리와 딥러닝 추론 병렬 처리 기반
Tensor Cores 32 Tensor Cores TensorRT 기반 FP16 추론 최적화에 활용

정리: Jetson Orin Nano는 상용 최종 장비를 확정했다는 의미가 아니라, 현장 Edge AI 처리 구조를 검증하기 위한 기준 장비이다. 상용 단계에서는 차선 수, 카메라 수, 해상도, 프레임레이트, 온도, 방진·방수, 장기 운영 안정성에 따라 Orin NX, AGX Orin, 산업용 Jetson 장비를 비교 검토해야 한다.

11.2 Zero-Copy / CUDA Memory / TensorRT 처리 전략

Edge Pipeline
GStreamer→ Zero-Copy 방향→ CUDA Memory→ TensorRT FP32→FP16→ Post Processing→ Passage Event

Zero-Copy의 정확한 의미

전체 프로그램의 모든 단계에서 복사가 0이라는 의미가 아니다. GStreamer/NVMM/CUDA Memory 경로를 활용해 입력·전처리·추론 구간의 CPU-GPU 메모리 이동을 최소화하는 구조이다.

TensorRT 적용 의미

YOLO 등 딥러닝 모델을 NVIDIA GPU에 맞게 최적화하고, FP32에서 FP16으로 경량화하여 반복 추론 지연을 줄이는 방향이다.

11.3 입력 영상 최적화

원본 입력

1920×1080 FHD 입력 프레임을 기준으로 차량·번호판 후보 영역을 탐색한다.

ROI Crop

번호판 관심 영역 중심으로 960×960 Crop을 적용한다. 전체 Resize가 아니라 필요한 영역을 집중 처리하는 전략이다.

파일 크기 예시

발표자료 기준 2.1MB 원본 입력을 0.9MB 수준의 최적화 영상으로 정리하는 예시를 제시했다.

데이터셋

AI Hub 번호판 데이터 9만 장 기반 학습 구조를 사용하며, 번호판 ROI 중심 학습·추론 전략을 적용한다.

핵심: 입력 최적화는 단순히 이미지를 줄이는 작업이 아니라, 번호판 인식에 필요한 정보를 보존하면서 불필요한 배경 연산을 줄이는 전략이다.

11.4 CRNN-OCR 채택 논리와 인식 타겟

CRNN-OCR 구조

CNN으로 이미지 특징을 추출하고, RNN/LSTM으로 문자 흐름을 처리한 뒤, CTC 방식으로 문자 시퀀스를 정렬한다.

인식 타겟

발표자료 기준 숫자 10개와 한글 40개, 총 50글자를 번호판 OCR 인식 대상으로 설정한다.

OCR 방식 특징 본 프로젝트 판단
CRNN CNN 특징 추출 + RNN/LSTM 시퀀스 처리 + CTC 정렬 가볍고 빠른 반복 추론에 적합하여 Edge 실시간 처리 기준으로 채택
Attention OCR 다중 줄·왜곡 텍스트 대응에 유리하지만 디코더 연산 부담 증가 향후 2줄 번호판, 왜곡 문자 대응 고도화 후보
Transformer OCR 문맥 기반 인식 성능 확장 가능성이 높지만 추론 비용 증가 상용 고도화 단계에서 성능·지연 비교 필요
VLM 문자 인식 외 장면 이해까지 확장 가능하나 Edge 비용이 큼 차세대 연구 과제로 분류

11.5 Best-Fit 산출 데이터 구조

누적 방식

차량이 ROI를 통과하는 동안 여러 프레임의 OCR 후보를 누적한다. 후보 수는 고정값보다 candidate_count와 agreement_ratio 기준으로 관리한다.

선정 기준

plate_confidence, 반복 등장 횟수, 후보 간 일치율, 번호판 형식 적합성을 기준으로 최종 plate_text를 선정한다.

검토 기준

신뢰도 또는 일치율 기준 미달 시 needs_review를 표시해 운영자 검토 흐름으로 연결한다.

필드 의미 활용
vehicle_pass_id 차량 통과 단위 식별자 OCR 후보와 통행 이벤트 연결
plate_text 최종 번호판 인식 결과 운영 조회 및 정산 후보 데이터의 핵심 값
plate_confidence 최종 결과 신뢰도 ACCEPT/REVIEW 판단 보조 지표
candidate_count 누적 OCR 후보 수 다중 프레임 누적 정도 확인
agreement_ratio 후보 간 결과 일치율 최종 결과 안정성 판단
needs_review 수동 검토 필요 여부 운영자 검토 및 이력 관리로 연결

11.6 WebTransport vs MQTT 전송 비교 및 상태 관리

전송 비교

프로젝트 성능 비교 기준으로 3000장 전송 테스트를 수행했고, WebTransport가 MQTT 대비 유선망에서 약 8배 높은 성능을 보인 것으로 정리했다.

LTE 환경

열악한 LTE 환경에서도 WebTransport 기반 구조가 더 나은 전송 성능을 보인 것으로 산출물에 반영한다.

보안 통신

QUIC과 TLS 1.3 기반 보안 통신 구조를 고려해 실시간 이벤트 전송의 안정성과 보안을 함께 다룬다.

전송 상태
SEND→ ACK 성공 수신 확인 RETRY 재전송 필요 REJECT 스키마/정합성 기준 거부

11.7 GPS 도로·구간 기반 / KPS 차선 기반 역할 구분

GPS: 도로·구간 기반 정합성

GPS는 현재 또는 Replay 기반 위치 데이터로 활용하며, 통행 이벤트가 특정 도로·구간·톨링 영역과 일치하는지 검증하는 데 사용한다.

KPS: 차선 단위 검증 가능성

KPS는 향후 국내 정밀 위치 인프라가 도입되었을 때, 도로 단위 검증을 넘어 차선 단위 위치 검증으로 확장하기 위한 KPS-ready 요소이다.

정확한 표현: GPS/KPS는 결제수단이 아니라 통행 이벤트의 위치 정합성을 검증하기 위한 데이터이다. GPS는 도로·구간 기반 검증, KPS는 향후 차선 단위 검증 가능성으로 구분한다.

11.8 관제 영상 표시와 이벤트 전송 분리

핵심 데이터

시스템의 핵심은 영상 스트리밍이 아니라 Jetson에서 생성한 통행 이벤트를 중앙 서버로 전송하고 검증하는 구조이다.

관제 영상

관제용 영상 표시는 SRT/HLS, MJPEG 등 운영 환경에 따라 선택 가능하다. Jetson Orin Nano 환경에서는 H.264 인코딩 부하와 장비 제약을 고려해 이벤트 전송과 분리 설계한다.
